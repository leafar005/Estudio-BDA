🗂️ 1. Catálogo del Sistema

1. Conceptos Clave

🤖

Catálogo del Sistema

Repositorio físico, de solo lectura, mantenido exclusivamente por el SGBD. Contiene metadatos puros (esquemas, restricciones, estadísticas del CBO) y es usado internamente por el motor.

👥

Diccionario de Datos

Repositorio conceptual, mantenido por humanos. Documenta el modelo del dominio, reglas de negocio y procesos asociados a la Base de Datos.

Uso Interno (Ejecución de Consultas)

Al ejecutar una consulta, el SGBD usa intensivamente el catálogo para tres fases:

  • Validar: Existencia de los objetos y privilegios del usuario.
  • Precompilar: Resolver léxicamente columnas (ej. expansión del comodín *).
  • Optimizar: Obtener estadísticas (bloques, cardinalidad, histogramas) para calcular el plan de menor coste en el CBO.

2. Estructura y Estándares

Estructura SQL Estándar

  • Cluster: Conjunto de catálogos accesibles por el usuario.
  • Catálogo: Colección de esquemas (debe incluir INFORMATION_SCHEMA).
  • Esquema: Colección de elementos (tablas, vistas, etc.).

Nomenclatura absoluta: catalogo.esquema.elemento

Implementaciones Físicas

Oracle: No sigue el estándar (Esquema = Usuario). Usa vistas jerárquicas.

PostgreSQL: El Catálogo equivale a la Database. Contiene esquemas nativos como pg_catalog y el estándar information_schema.

Vistas Clave del Catálogo Oracle

USER_TABLES / TABS USER_TAB_COLUMNS / COLS USER_OBJECTS / OBJ USER_CONSTRAINTS USER_SYS_PRIVS

Regla de prefijos de Oracle: USER_ (lo propio del usuario), ALL_ (a lo que el usuario tiene acceso), DBA_ (todo el sistema, solo administradores).

Consulta al Catálogo
-- Consultar todas las columnas de la tabla empleado
SELECT table_name, column_name, data_type 
FROM information_schema.columns 
WHERE table_schema = 'public' 
  AND table_name = 'empleado';

3. Conceptos Avanzados de Catálogo

🌟 Concepto Nivel Pro: Histogramas en el Catálogo
El CBO no solo necesita saber cuántas filas hay, sino cómo se distribuyen. Si una columna está sesgada (ej. 95% de los clientes son de "Madrid"), el catálogo almacena un Histograma. Gracias a él, el CBO sabe que buscar "Madrid" requiere un Full Scan (porque es casi toda la tabla), pero buscar "Soria" justifica usar el índice.

👁️ 2. Vistas

1. Conceptos Clave

Definición y Arquitectura

Una vista es una "tabla virtual" que expone datos de consultas sobre otras tablas. Ayudan a conseguir independencia lógica y definen los esquemas externos en la arquitectura ANSI/SPARC.

Almacenamiento

Solo se almacena su definición precompilada en el catálogo, no sus datos. La excepción son las Vistas Materializadas que sí guardan datos físicos en disco.

2. Actualización y Tuplas Migratorias

Vistas Actualizables

Para permitir DML, el SGBD debe trazar la ruta de vuelta a la tabla base unívocamente. No son actualizables si tienen: DISTINCT, GROUP BY, HAVING, operaciones de conjuntos, o ciertos JOINs.

Tuplas Migratorias

Filas que "desaparecen" de la vista tras un INSERT/UPDATE porque dejan de cumplir la condición (WHERE) de la vista. Se evita su aparición usando WITH CHECK OPTION.

Tipos de CHECK OPTION

  • CASCADED (Defecto): Comprueba la condición de la vista y de todas sus vistas base.
  • LOCAL: Sólo comprueba la restricción de la vista actual.
Vista con CHECK OPTION
-- Crear vista que bloquea inserciones de empleados fuera de IT
CREATE VIEW v_empleados_it AS 
SELECT empno, nombre, salario, dpto 
FROM empleado WHERE dpto = 'IT' 
WITH CHECK OPTION;

-- Este INSERT fallará porque incumple el WHERE (Tupla Migratoria)
INSERT INTO v_empleados_it VALUES (10, 'Ana', 2000, 'HR');

3. Oracle y Vistas Materializadas

Particularidades de Oracle

Permite ORDER BY (el estándar no) y la restricción WITH READ ONLY. El CHECK OPTION actúa siempre en cascada.
Nota: ALTER VIEW nombre COMPILE; solo valida la vista, no cambia su definición (se usa CREATE OR REPLACE).

Vistas Materializadas

Aumentan rendimiento guardando filas reales (instantáneas) para DWH.
BUILD: IMMEDIATE, DEFERRED.
REFRESH: COMPLETE, FAST (incremental por logs), FORCE.

👑
Query Rewrite (Truco Optimización): El optimizador de Oracle puede usar vistas materializadas internamente mediante Query Rewrite, redirigiendo tu consulta hacia la instantánea precalculada aunque tú hayas hecho explícitamente el SELECT a las tablas base.

🛡️ 3. Integridad

1. Conceptos Claves

Definición de Integridad

La información almacenada debe ser Correcta (evitar información errónea, ej. edad negativa) y Completa (toda información requerida debe estar presente).

Mecanismos de Garantía

SGBD Transaccional (ACID) Restricciones Declarativas Triggers (Reglas ECA)

2. Restricciones en SQL

PK & UNIQUE

PK: No nulos, no duplicados. Solo 1 por tabla.

UNIQUE: No duplicados. Puede haber varias y admiten nulos.

NOT NULL & CHECK

NOT NULL: Solo a nivel atributo.

CHECK: Establece valores válidos. Limitación: no admite subconsultas reales en ningún SGBD.

FOREIGN KEY & MATCH

Garantiza integridad referencial apuntando a PK/UNIQUE. Si la FK es compuesta y tiene NULLs, actúa la cláusula MATCH:

MATCH SIMPLE (Por defecto): Válido si algún componente es NULL.
MATCH FULL: Válido solo si todos son NULL o ninguno lo es.
MATCH PARTIAL: Válido si hay correspondencia parcial en la tabla referenciada.

3. Restricciones Aplazadas (DEFERRABLE)

Posponen la comprobación al final de la transacción (COMMIT). Si falla, hace ROLLBACK total. Útiles para ciclos referenciales y cargas masivas.

INITIALLY DEFERRED
Inicia aplazada para toda la transacción.
INITIALLY IMMEDIATE
Inicia inmediata pero permite modificarse en vuelo con SET CONSTRAINTS.
Acciones Referenciales
-- Crear tabla con Foreign Key y acciones referenciales
CREATE TABLE empleado (
    id INT PRIMARY KEY,
    dept_id INT,
    CONSTRAINT fk_dept FOREIGN KEY (dept_id) 
    REFERENCES departamento(id) 
    ON DELETE RESTRICT      -- Inmediato: aborta si hay empleados (NO aplazable)
    ON UPDATE NO ACTION     -- Idéntico a RESTRICT, pero SÍ es aplazable
    DEFERRABLE INITIALLY DEFERRED -- Aplaza la comprobación de NO ACTION al COMMIT
);

🔐 4. Seguridad en SGBD

1. Conceptos Fundamentales

🛡️

Propiedades CIA

La base de la seguridad: Confidencialidad (datos privados), Integridad (sin modificación indebida) y Disponibilidad (acceso garantizado).

📋

SGSI e ISO 27001

Metodología basada en identificar activos, amenazas y evaluar el riesgo mediante el ciclo PDCA de mejora continua.

🔎

Auditoría FGA

Fine Grain Auditing permite auditar operaciones de lectura pura (SELECT) implementando políticas granulares a nivel de fila y columna.

2. Modelos de Control de Acceso

DAC (Discrecional)

El creador decide mediante GRANT/REVOKE. Permite delegación (WITH GRANT OPTION). La revocación de objetos se propaga en cascada recursivamente.

MAC (Obligatorio)

Políticas del sistema (niveles de seguridad). Bell-LaPadula: No leer hacia arriba (Prop. Simple), No escribir hacia abajo (Prop. Estrella).

RBAC (Roles) 🚨

Estándar en SGBD modernos. Los permisos se asignan a roles (perfiles) y estos a usuarios. Simplifica drásticamente la administración frente al caos de cadenas del DAC puro.

Gestión de Roles y Privilegios
-- 1. Crear un rol
CREATE ROLE admin_ventas;

-- 2. Otorgar permisos sobre un objeto (Tabla) al rol
GRANT SELECT, INSERT, UPDATE ON ventas TO admin_ventas;

-- 3. Asignar el rol a un usuario
GRANT admin_ventas TO juan;

-- 4. Dar permiso con derecho a delegar (WITH GRANT OPTION)
GRANT DELETE ON ventas TO maria WITH GRANT OPTION;

3. Técnicas Adicionales y Auditoría

  • Inyección SQL: Vulnerabilidad por no sanear entradas. Prevención: Uso estricto de Bind Parameters (parámetros vinculados).
  • Cifrado TDE: Transparent Data Encryption cifra los datos físicos en disco bidireccionalmente. Las contraseñas, en cambio, usan Hashes unidireccionales.
  • Bases de Datos Estadísticas: Protegen la privacidad publicando solo agregados (SUM, AVG) y limitando respuestas a grupos pequeños para evitar inferencia.
  • Auditoría Nativa vs Triggers: La nativa registra eventos DDL o accesos genéricos. Los Triggers de Sistema capturan eventos globales ajenos a tablas, como AFTER LOGON ON DATABASE.

⚡ 5. Reglas Activas y Triggers

1. BD Pasivas vs Activas

Las pasivas requieren gestión explícita. Las activas reaccionan automáticamente a eventos (DML) usando Triggers para controlar restricciones de negocio complejas o realizar auditorías en la sombra.

La Arquitectura ECA

E - Evento (INSERT/UPDATE) C - Condición (WHEN) A - Acción (PL/SQL)

2. Granularidad y Variables de Transición

Tiempos y Tipos

  • Tiempos: BEFORE, AFTER, INSTEAD OF (solo vistas).
  • Granularidad: FOR EACH ROW (por fila) o STATEMENT (por sentencia, defecto).
  • Diferenciador DML: INSERTING, UPDATING, DELETING.

Variables :OLD y :NEW

Exclusivas de FOR EACH ROW.
⚠️ Clave de Examen: :OLD nunca es modificable; :NEW sólo puede ser modificado en triggers BEFORE.

3. Errores Típicos y Soluciones Modernas

Tabla Mutante (ORA-04091)

Ocurre cuando un trigger a nivel de fila intenta leer/modificar (SELECT/UPDATE) la misma tabla que lo está disparando.

Solución: Triggers Compuestos 🚨

(Oracle 11g+). Permiten agrupar tiempos STATEMENT y ROW en un solo bloque con estado de memoria compartido, esquivando la tabla mutante elegantemente.

* Nota: Los triggers INSTEAD OF son la única forma de hacer actualizables las vistas complejas (con JOINs o GROUP BY).

Estructura de un Trigger
-- EVENTO: ANTES de actualizar el salario
CREATE OR REPLACE TRIGGER trg_auditoria_salario
BEFORE UPDATE OF salario ON empleado
FOR EACH ROW
-- CONDICIÓN: Solo si la subida es mayor al 50%
WHEN (NEW.salario > OLD.salario * 1.5)
-- ACCIÓN:
BEGIN
    INSERT INTO log_alertas (emp, old_sal, new_sal, fecha)
    VALUES (:OLD.empno, :OLD.salario, :NEW.salario, SYSDATE);
END;

💾 6. Diseño Físico

1. Organización Física (La Independencia Física)

El usuario consulta el nivel lógico (tablas) sin preocuparse del físico (ficheros).
Jerarquía: Tablespace → Datafiles → Segmentos → Extensiones → Bloques (unidad mínima I/O, ej: 8KB).

🥞

Heap (Apilada)

Por defecto. Inserción rápida (donde haya hueco), pero exige Full Table Scan sin índices.

🌲

IOT

La tabla ES el índice (B+). Datos ordenados por PK. Lectura brutal por PK, escritura lenta.

🔗

Cluster

Mezcla filas de distintas tablas en el mismo bloque si comparten clave (B+ tree o Hash). JOINs instantáneos.

2. Índices Core

Árbol B+ y Complejidad

Para alta cardinalidad (ej. DNI). Nodos hoja con pares Clave-ROWID enlazados. Asegura búsquedas logarítmicas O(log n). Penaliza escrituras DML.

Bitmap (Mapa de Bits)

Para baja cardinalidad (ej. Sexo) en Data Warehouses. Muy rápidos para cruces AND/OR, pero sufren bloqueos masivos ante modificaciones concurrentes.

Gestión de Nulos en Árboles B+ (Trampa de Examen)

Regla de Oro en Oracle: Los B-Tree convencionales NUNCA almacenan filas donde todos los atributos indexados son NULL.
• Funciones como MAX(), MIN() o consultas IS NULL NO usarán el índice si no se garantiza la exclusión de nulos (ej. falta de un constraint NOT NULL).
• Si el índice es compuesto, la fila se indexa si al menos un componente es NO nulo.

3. Índices Avanzados y Particionado

  • Índices de Clave Inversa (Reverse Key): Invierten los bytes (ej. 1234 → 4321). Obligatorios para columnas alimentadas por SEQUENCE para evitar la contención (cuello de botella) en el mismo nodo hoja. No soportan búsquedas por rangos.
  • Basados en Funciones: Indexan el resultado precalculado de una expresión, como UPPER(nombre).
  • Particionado de Índices: Al fragmentar una tabla masiva (>1GB) por rangos, listas o hash, el índice puede ser Local (se divide igual que la tabla) o Global (sigue su propia regla). Mayor ventaja: Partition Pruning.

⚙️ 7. Optimización de Consultas

1. Fases de Ejecución

1. Transformación

Paso de SQL a un Árbol o Grafo de Álgebra Relacional Extendida.

2. Opt. Heurística

Aplicación de equivalencias algebraicas para reducir el tamaño de datos intermedios (Forma Canónica).

3. Opt. Costes (CBO)

Usa el catálogo (bloques, histogramas) para estimar I/O y elegir el plan físico óptimo.

2. Procedimientos Físicos (Acceso y Joins)

Acceso de Bajo Nivel (CBO)

  • TABLE ACCESS FULL: Escaneo completo. Usado sin índices o con baja selectividad.
  • INDEX UNIQUE SCAN: Devuelve exactamente 1 fila (por PK/Unique).
  • INDEX RANGE SCAN: Búsqueda de rangos (>, <) o desigualdades.
  • INDEX FULL SCAN: Lee todo el índice ordenadamente. Evita el TABLE ACCESS FULL si todos los datos requeridos están en el índice.
  • INDEX FAST FULL SCAN: Lee todo el índice ignorando el orden, usando multi-block I/O.
  • INDEX SKIP SCAN: Filtra por columnas secundarias de un índice compuesto saltándose la primera.

Algoritmos de Joins Físicos

  • Nested Loop: Itera externa contra interna. Excelente si la interna tiene índice. Vale para cualquier condición.
  • Sort-Merge: Ordena ambas y mezcla linealmente. Solo para equi-joins. El rol externa/interna da igual.
  • Hash Join: Aplica hash a la tabla pequeña en RAM, y escanea la grande. Ideal sin índices (solo equi-joins).

🚑 8. Recuperación ante Fallos

1. Conceptos Clave y Garantías

Garantía ACID

Atomicidad y Durabilidad: Los COMMIT sobreviven, los abortos (ROLLBACK) deshacen lo escrito en disco mediante operaciones UNDO.

Idempotencia

Aplicar redo(redo(Ti)) da el mismo resultado que redo(Ti). Vital por si ocurre una segunda caída eléctrica durante la propia recuperación.

Protocolo WAL (Write Ahead Logging):

Obliga a escribir los búferes de log en el disco físico antes que los búferes de datos. Sin esto, la base de datos se corrompería irrevocablemente ante cortes de luz.

2. Dinámica UNDO/REDO y Algoritmo ARIES

Algoritmo ARIES 🚨

Es el estándar de recuperación de la industria (basado en WAL y la política Steal/No-Force). Consta de 3 fases obligatorias:

  • 1. Análisis: Lee desde el último Checkpoint para saber qué transacciones estaban activas y qué bloques estaban sucios al caer el sistema.
  • 2. REDO (Repetir la historia): Aplica todos los cambios del log hacia adelante para dejar la memoria RAM tal cual estaba un milisegundo antes de caer.
  • 3. UNDO (Deshacer): Recorre el log hacia atrás revirtiendo las transacciones que no lograron hacer COMMIT.

3. Archivos de Log: Actualización Diferida vs Inmediata

Actualización Diferida (Deferred)

Los cambios NO van a disco hasta el COMMIT. Si falla, no hay nada que deshacer.
• Formato Log: [Ti, Registro, Valor_Nuevo]
• Recuperación: Solo requiere aplicar REDO a las confirmadas. No existe fase UNDO.

Actualización Inmediata

Los cambios pueden ir a disco antes del COMMIT (política Steal). Exige poder deshacer.
• Formato Log: [Ti, Registro, Valor_Viejo, Valor_Nuevo]
• Recuperación: Requiere tanto UNDO (para abortadas) como REDO (para confirmadas).

4. Transacciones Distribuidas

Protocolo 2PC 🚨

El Two-Phase Commit Protocol asegura ACID en nodos distribuidos guiado por un Coordinador:
Fase 1 (Prepare): Pregunta a todos los nodos si están listos para confirmar y asegurar en log.
Fase 2 (Commit): Si TODOS dicen sí, manda el COMMIT global. Si uno solo falla, manda ROLLBACK global.

🚦 9. Concurrencia y Transacciones

1. Problemas Clásicos

Actualización Perdida

Dos escriben a la vez, gana el último.

Lectura Sucia

Leer un dato de otra Tx que aún no ha hecho commit.

Lectura No Repetible

El valor muta entre dos SELECTs de la misma Tx.

Fantasmas

Aparecen filas nuevas en un rango de un SELECT.

2. Bloqueos de Intención (Granularidad Múltiple) 🚨

Para bloquear a nivel de fila y a la vez permitir verificar colisiones a nivel de tabla sin escanearla entera, se usan los bloqueos de intención aplicados desde la raíz hacia abajo:

  • IS (Intention Shared): Aviso de que la Tx planea leer (poner S) a nivel inferior (ej. fila).
  • IX (Intention Exclusive): Aviso de que la Tx planea escribir (poner X) a nivel inferior. Evita que otra Tx bloquee la tabla completa.
  • SIX (Shared + Intention Exclusive): Lee toda la tabla, pero solo modificará algunas filas inferiores.

3. Ordenación por Marcas de Tiempo y MVCC 🚨

Regla de Escritura de Thomas

En el protocolo de Marcas de Tiempo (Timestamp Ordering), si una transacción $T_i$ intenta escribir un dato que ya fue escrito en el futuro por una $T_j$ ($M(T_i) < ME$), la escritura simplemente se ignora y $T_i$ sobrevive. No se hace abort ni Rollback.

Multiversión (MVCC)

Mantiene múltiples versiones de un dato. Para leer, se busca la versión con la mayor $ME \le M(T_i)$. Para escribir, si se intenta modificar una versión que ya ha sido leída por alguien en el "futuro" ($M(T_i) < ML_{versión}$), la transacción aborta.

3. Protocolos 2PL y Deadlocks

2PL Estricto / Riguroso

  • 2PL: Crecimiento (adquiere) y decrecimiento (libera). Garantiza seriabilidad.
  • Estricto: Retiene bloqueos exclusivos (X) hasta el final (commit/rollback).
  • Riguroso: Retiene TODOS (S y X) hasta el final. Evitan anulación en cascada.

Prevención de Deadlocks

  • Wait-Die: Prioriza la vejez. Si el que pide es viejo, espera. Si es joven, muere (aborta).
  • Wound-Wait: Expropiación agresiva. Si el que pide es viejo, hiere (aborta al joven). Si es joven, espera.
