📊 1. CLASIFICACIÓN DE EJERCICIOS

Tras diseccionar los exámenes, los problemas se dividen en 6 grandes familias.

Tipo A: Vistas y Seguridad

Qué se pide: Indicar qué pasa si se consulta una vista tras modificar la tabla base, o si se intentan insertar filas migratorias. Privilegios y roles.

⚠️ Trampa principal:
El SELECT * al crear una vista se "congela" en el catálogo. Un ALTER TABLE posterior añadiendo una columna no aparecerá en la vista. Además, PUBLIC es un pseudo-rol que no puede ser revocado a un único usuario para aislarlo.

Tipo B: Integridad (Cláusula MATCH y Acciones Referenciales)

Qué se pide: Validar tuplas candidatas frente a una Foreign Key compuesta con nulos. Diferenciar NO ACTION de RESTRICT.

⚠️ Trampa principal:
MATCH FULL rechaza cualquier registro "a medias" (mezcla de nulos y no nulos). NO ACTION permite aplazamiento (DEFERRABLE), mientras que RESTRICT es siempre de verificación inmediata.

Tipo C: Optimización Heurística (Álgebra Relacional)

Qué se pide: Dibujar el árbol canónico y el árbol optimizado justificando los pasos a partir de una consulta SQL.

Conceptos evaluados: Equivalencias del álgebra relacional, empujar selecciones (Push-Down), convertir productos cartesianos en Joins.

⚠️ Trampa principal:
Olvidar el operador de agrupación (γ o gamma) en consultas con GROUP BY o funciones de agregación (SUM, COUNT).

Tipo D: Diseño Físico (Índices y Planes de Ejecución)

Qué se pide: Predecir si Oracle usará o no un índice dado, o analizar la salida de un EXPLAIN PLAN.

Conceptos evaluados: Índices B-Tree, comportamiento ante valores nulos, lectura de grafos de ejecución.

⚠️ Trampa principal:
Los índices B-Tree de Oracle NO guardan valores completamente nulos. Si una query hace un SELECT columna o un COUNT(columna) sin un IS NOT NULL, el optimizador descartará el índice y hará un TABLE ACCESS FULL. Excepción: Funciones como MAX() o AVG() sí usan el índice porque ignoran los nulos matemáticamente.

Tipo E: Recuperación ante Caídas (Logs y Checkpoints)

Qué se pide: Dado un archivo de log y una caída del sistema, deducir la arquitectura y listar qué transacciones sufren UNDO y cuáles REDO.

Conceptos evaluados: Protocolo WAL, Actualización Diferida vs. Inmediata, y Puntos de Verificación (Checkpoints).

⚠️ Trampa principal:
La barrera del CHECKPOINT. Si una transacción hizo COMMIT antes de un checkpoint, se ignora por completo en la recuperación, ya que sus datos ya están seguros en disco. Otra trampa es confundir la arquitectura: si el log tiene 4 parámetros (Tx, Dato, ValorViejo, ValorNuevo) es Inmediata; si tiene 3, es Diferida. En Diferida, el UNDO no existe.

Tipo F: Control de Concurrencia (Marcas de Tiempo y MVCC)

Qué se pide: Trazar el estado de las variables (Marcas de Lectura ML y Marcas de Escritura ME) tras una secuencia de operaciones Read(X) / Write(X).

Conceptos evaluados: Regla de Escritura de Thomas y Control Multiversión (MVCC).

⚠️ Trampa principal:
Los abortos falsos en la Regla de Thomas. Si una transacción intenta escribir un dato que ya fue escrito por una transacción más moderna (del "futuro"), ¡NO se aborta! Se ignora silenciosamente. En MVCC, la trampa es no elegir correctamente la versión base sobre la que se opera (se debe buscar la que tenga el mayor ME ≤ M(T)).

🛠️ 2. MÉTODO GENERAL DE RESOLUCIÓN (Tus Recetas)

Receta 1: Optimización Heurística (Álgebra)

Procedimiento Universal:
1
Árbol Bruto: Junta todas las tablas con producto cartesiano ($\times$) en la base. Pon una super-selección ($\sigma$) encima con TODO el WHERE. Remata con la agrupación ($\gamma$) y la proyección ($\pi$).
2
Push-Down (Bajar Filtros): Rompe la super-selección en condiciones atómicas y bájalas todo lo posible hasta pegarlas a sus tablas respectivas.
3
Joins: Convierte las parejas (Tabla $\times$ Tabla) + Selección de igualdad en operadores Join ($\bowtie$).
4
Proyecciones Intermedias: Pon operadores $\pi$ antes de cada Join para eliminar columnas inútiles.

Receta 2: Logs de Recuperación

Cómo reconocerlo: Lista de comandos <Tx, START>, <Tx, A, 10, 20>.

Procedimiento Universal:
1
Clasificar Arquitectura: Si ves 4 campos en el DML (Transacción, Variable, Antiguo, Nuevo) es Inmediata. Si ves 3 (Transacción, Variable, Nuevo) es Diferida.
2
Barrera Checkpoint: Traza una línea en el último <CHECKPOINT (activas)>. Tacha cualquier transacción que tenga un <COMMIT> por encima de esa línea. Ya no nos importan.
3
Clasificar Transacciones Restantes:
  • Tienen <COMMIT> → Van a la lista de REDO.
  • No tienen <COMMIT> → Van a la lista de UNDO (sólo si es arquitectura Inmediata).
4
Ejecución: Aplica primero UNDO (de abajo hacia arriba en el log) y luego REDO (de arriba hacia abajo).

Receta 3: Concurrencia con MVCC (Multiversión)

Cómo reconocerlo: El enunciado te da versiones como A1: Valor=20, ML=4, ME=1.

Procedimiento Universal:
1
Encontrar la versión: Ante una operación de la transacción $T_i$, busca la versión histórica que tenga el mayor ME $\le M(T_i)$. Esa es tu base.
2
Si es Lectura (Read): Se permite siempre. Actualiza el $ML$ de esa versión: $ML = \max(M(T_i), ML actual)$.
3
Si es Escritura (Write): Comprueba dos reglas sobre esa versión base:
  • ¿$M(T_i) < ML_{base}$? Si SÍ → ROLLBACK (alguien del futuro ya lo leyó).
  • ¿$M(T_i) < ME_{base}$? Si SÍ → ROLLBACK (alguien del futuro ya lo sobreescribió).
  • Si pasa ambas → Crea una nueva versión $A_{nueva}$ con $Valor = Nuevo$, $ME = M(T_i)$, $ML = M(T_i)$.

Examen 11-07-2022 | P2 (0,5p). Seguridad y Revocación de Roles

Enunciado:
Di qué pasaría si un usuario con el rol ABD ejecutase la siguiente sentencia SQL en Oracle. Específicamente, ¿qué ocurre con los usuarios que ya están conectados?

SQL
REVOKE CONNECT FROM PUBLIC;
🎯 Resolución:
Causaría la revocación del rol CONNECT (el cual posee el privilegio CREATE SESSION necesario para abrir una sesión contra la base de datos) a absolutamente todos los usuarios, dado que PUBLIC es un pseudo-AuthID que engloba a todos los usuarios del sistema (presentes y futuros).

Este efecto no sería instantáneo para los usuarios ya conectados, dado que la revocación de un rol no se hace efectiva hasta que se inicia una nueva sesión (CONNECT usr/key) o se ejecuta explícitamente un SET ROLE ALL. Afectará, por lo tanto, a las sesiones futuras y a las presentes en cuanto intenten revalidar sus roles.

Examen Enero 2026 (Parcialmente) | P2 (0,5p). SQL Estándar y Diseño Relacional

Enunciado:
Utilizando SQL estándar: ¿El hecho de que una vista tenga la cláusula WITH CHECK OPTION garantiza que no se pueda insertar una fila migratoria a través de ella? Justifica la respuesta.
🎯 Resolución:
Solución:

Sí, en términos predeterminados lo garantiza, pero hay excepciones dependiendo de su especificación jerárquica explícita. En el estándar SQL, la inclusión de WITH CHECK OPTION evita directamente la aparición de "tuplas migratorias", es decir, impide insertar o actualizar filas que violen la condición filtrante y desaparezcan del resultado de la vista.

Justificación detallada:

El SQL asume por defecto un alcance en cascada (CASCADED CHECK OPTION) si sólo se escribe WITH CHECK OPTION. Bajo el modelo CASCADED, el sistema comprueba tanto la condición de la propia vista como las de todas las vistas base sobre las que pudiera estar anidada, garantizando absoluta invulnerabilidad frente a filas migratorias.

Sin embargo, no lo garantiza de manera absoluta en toda la jerarquía si el desarrollador especifica explícitamente WITH LOCAL CHECK OPTION sobre una vista construida a partir de otra vista. Bajo la modalidad LOCAL, la base de datos solo comprobará la condición de la vista actual (la más superficial) de forma restrictiva. Si se insertase una fila que cumple con la restricción local, pero que no cumple con el filtro original de la vista subyacente, la inserción se permitiría temporalmente pero la fila migraría igualmente y no sería visible, rompiendo la garantía de la vista base.

Indica, para cada posible especificación de la cláusula MATCH de la clave foránea, qué filas de ESCLAVA serían válidas.
🎯 Resolución:
• MATCH SIMPLE (Por defecto): Son válidas las filas (1, 2, 3).

Justificación: Requiere que todos los componentes sean completamente NULL o que ninguno lo sea y verifiquen la clave en MAESTRA. Las filas 1 y 2 mezclan valores con nulos, por lo que quedan invalidadas.

Examen 06-07-2022 | P6b (0,25p). Restricciones Aplazables

Enunciado:
¿Qué es una restrición aplazable? Pon un ejemplo donde sea necesario su uso.
🎯 Resolución:
Una restricción aplazable es aquella definida con la opción DEFERRABLE, donde la comprobación de integridad no se realiza inmediatamente después de ejecutar la instrucción DML correspondiente, sino que se pospone hasta el momento de confirmar la transacción (COMMIT). Si en el momento del COMMIT la restricción falla, la base de datos realiza un ROLLBACK completo de toda la transacción.

Ejemplo de uso clásico: Es indispensable cuando existen ciclos referenciales (claves foráneas cruzadas) con restricciones de tipo NOT NULL entre dos o más tablas. Sin aplazar las restricciones, no se podría realizar ninguna inserción inicial en ninguna tabla, ya que cualquier registro violaría instantáneamente la foreign key de la otra.

Examen 11-07-2022 | P6b (0,25p). Restricciones Aplazables (Deferrable)

Enunciado:
¿Qué es una restrición aplazable? Pon un ejemplo donde sea necesario su uso.
🎯 Resolución:
Las restricciones aplazadas son aquellas definidas con la opción DEFERRABLE. Al contrario que las normales (NOT DEFERRABLE), que se comprueban de manera inmediata tras lanzar cada query, con las aplazables podemos modificar el comportamiento con la sentencia:

SQL
SET CONSTRAINTS { ALL | restr1, ... } { IMMEDIATE | DEFERRED };


• IMMEDIATE: Se comporta en el momento, de manera inmediata.
• DEFERRED: Esperará hasta el momento de confirmar la transacción (COMMIT) para comprobar la validez de la integridad.


Ejemplo de uso: Es totalmente necesario para romper ciclos referenciales entre dos tablas (por ejemplo, cuando la tabla A tiene una foreign key no nula hacia la tabla B, y la tabla B tiene otra foreign key no nula hacia la tabla A). Sin aplazar la restricción, sería imposible insertar el primer registro en ninguna de ellas.

Examen 2023 | P4. Acciones Referenciales (`ON DELETE / ON UPDATE NO ACTION`)

Enunciado:
Explica el funcionamiento y la semántica de la acción referencial NO ACTION. Puedes ayudarte de un ejemplo para completar tu explicación.
🎯 Resolución:

Respondido por el alumno:



La cláusula NO ACTION especifica que si los cambios que se pretenden realizar sobre una clave primaria o única violan o rompen la integridad referencial, la operación de borrado o actualización no se ejecutará y el motor de la base de datos devolverá un error. 6. Recuperación ante Caídas (Logs y Checkpoints)

Examen 06-07-2022 | P2 (0,5p). Protocolo WAL (Write Ahead Logging)

Enunciado:
¿Qué es el protocolo WAL (write ahead logging)? Explica cómo funciona y por qué es necesario.
🎯 Resolución:
Es un protocolo de escrituras anticipadas que especifica que siempre se debe registrar el cambio en el fichero de log antes de que se vea reflejado en la base de datos física.

Funcionamiento: Modifica el buffer de log antes que el buffer de datos. Cuando se va a volcar un buffer de datos a disco, antes deben volcarse de forma obligatoria los registros del buffer de log relacionados con dicha transacción al almacenamiento estable (fichero de log).

Por qué es necesario: Porque asegura que se guarde la información de cómo rehacer (redo) o deshacer (undo) las transacciones antes de consolidar los cambios. Si se volcasen a disco los buffers de datos antes y se produjese un fallo de energía o caída del sistema antes de escribir el log, no seríamos capaces de recuperar la consistencia de las transacciones.

8. $T_1$: Write(C) $→$ FALLO (Rollback de $T_1$). Como $M(T_1) = 1 < ML(C) = 3$, $T_1$ intenta escribir un registro que ya ha sido leído por una transacción más reciente ($T_3$). $T_1$ se aborta y se reiniciará.

Examen Enero 2026 (Parcialmente) | P6 (0,75p). Concurrencia y Transacciones

Enunciado:
Considera un sistema con control de concurrencia multiversión. La marca de transacción será su subíndice, esto es, para cada transacción $T_i$, $M(T_i)=i$. Para el dato A tenemos las siguientes versiones:

$A_1 = 10$$\text{ ML} = 5$
$\text{ME} = 2$
$A_2 = 20$$\text{ ML} = 9$
$\text{ME} = 7$
Indica lo que pasa en cada una de las siguientes operaciones (si puede hacerse o no, qué versión se lee o escribe, etc.) y cómo queda cada una de las versiones de A.

1. $T_4$ lee A.
🎯 Resolución:
• Efecto: Como $M(T_4) = 4$, la transacción busca la versión más reciente cuya marca de escritura (ME) sea menor o igual a 4, que corresponde a $A_1$ ($ME = 2$). Dado que $M(T_4) = 4$ no supera a la marca de lectura (ML) actual de esa versión ($ML = 5$), se realiza la lectura devolviendo el valor 10, pero sin modificar las marcas, actualizándose como $\max(4, 5) = 5$.
• Estado de versiones:
- $A_1 = 10 → ME = 2, ML = 5$ - $A_2 = 20 → ME = 7, ML = 9$
1. $T_9$ lee A.
[!success]- Solución

• Efecto: Como $M(T_9) = 9$, la versión adecuada es la más reciente con $ME \le 9$, que es $A_2$ ($ME = 7$). Como $M(T_9) = 9$ se compara con el $ML(A_2) = 9$, el sistema asume $\max(9, 9) = 9$, por lo que $T_9$ lee el valor 20 y las marcas permanecen inalteradas.
• Estado de versiones:
- $A_1 = 10 → ME = 2, ML = 5$ - $A_2 = 20 → ME = 7, ML = 9$
1. $T_{13}$ escribe 13 en A.
[!success]- Solución

• Efecto: Al ser $M(T_{13}) = 13$, intenta escribir sobre la base de la versión más reciente con $ME \le 13$, que es $A_2$ ($ME = 7$). Se verifica si el dato fue leído en el "futuro" respecto a $T_{13}$ comprobando si $M(T_{13}) < ML(A_2)$; dado que $13 < 9$ es falso, se permite la escritura. $T_{13}$ crea una nueva versión $A_3 = 13$ y se le asignan las marcas $ML = 13$ y $ME = 13$.
• Estado de versiones:
- $A_1 = 10 → ME = 2, ML = 5$ - $A_2 = 20 → ME = 7, ML = 9$ - $A_3 = 13 → ME = 13, ML = 13$
1. $T_4$ escribe 25 en A.
[!success]- Solución

• Efecto: Como $M(T_4) = 4$, la versión correspondiente en su línea temporal es $A_1$ ($ME = 2$). Se evalúa si la marca temporal de $T_4$ es menor a alguna lectura realizada en el "futuro" ($ML(A_1) = 5$); como $4 < 5$ es verdadero, la operación se deniega para evitar anomalías (evitar modificar un dato que una transacción más moderna ya procesó). $T_4$ hace rollback y se reiniciará con una nueva marca temporal.
• Estado de versiones: (No cambian)
- $A_1 = 10 → ME = 2, ML = 5$ - $A_2 = 20 → ME = 7, ML = 9$ - $A_3 = 13 → ME = 13, ML = 13$
2. $T_6$ escribe 50 en A.
[!success]- Solución

• Efecto: Como $M(T_6) = 6$, busca la versión aplicable $ME \le 6$, que es $A_1$ ($ME = 2$). Se evalúa la validez con el $ML(A_1) = 5$; dado que $M(T_6) \ge ML(A_1)$ ($6 \ge 5$), se permite la operación. Se crea una nueva versión con el valor 50. Dado que esto ocurre en un instante temporal entre $A_1$ y $A_2$, se genera una versión intermedia válida (se reajustan las posiciones por marcas de escritura) con marcas $ME = 6$ y $ML = 6$.
• Estado de versiones:
- $A_1 = 10 → ME = 2, ML = 5$ - $A_{intermedia} = 50 → ME = 6, ML = 6$ - $A_2 = 20 → ME = 7, ML = 9$ - $A_3 = 13 → ME = 13, ML = 13$
